# Паттерн Bridge в системе уведомлений

## Что такое Bridge

`Bridge` или «Мост» — это структурный паттерн проектирования, который разделяет абстракцию и реализацию так, чтобы их можно было изменять независимо друг от друга.

Он применяется в тех случаях, когда в программе есть две самостоятельные группы изменений. Вместо создания большого количества классов под каждую возможную комбинацию, эти группы выносятся в отдельные иерархии и связываются через композицию.

## Почему этот паттерн подходит здесь

В данной программе есть две независимые части:

- тип уведомления
- способ отправки уведомления

Тип уведомления отвечает на вопрос: что именно мы отправляем.  
Способ отправки отвечает на вопрос: как именно мы это отправляем.

Если не использовать `Bridge`, пришлось бы создавать отдельные классы для каждой комбинации, например: обычное уведомление по Email, обычное уведомление по SMS, срочное уведомление по Push и так далее. При увеличении числа типов уведомлений и каналов количество классов быстро росло бы.

Паттерн `Bridge` решает эту проблему, потому что позволяет комбинировать любой тип уведомления с любым каналом отправки.

## Где паттерн реализован в программе

Абстракция представлена базовым классом `Notification`. Он задает общую модель уведомления и хранит ссылку на объект канала отправки.

Уточненные абстракции:

- `BasicNotification`
- `UrgentNotification`
- `ConfirmationNotification`

Они определяют конкретные виды уведомлений и логику формирования сообщения.

Реализация представлена интерфейсом `INotificationChannel`.

Конкретные реализации:

- `EmailChannel`
- `SmsChannel`
- `PushChannel`

Они определяют конкретный способ отправки уведомления.

## Как это работает

Каждый объект уведомления получает канал отправки через конструктор. Это значит, что уведомление не зависит от конкретной реализации канала, а работает через общий интерфейс.

Например, можно создать:

- обычное уведомление с отправкой по Email
- срочное уведомление с отправкой по SMS
- уведомление с подтверждением с отправкой через Push

При этом сами классы уведомлений не нужно переписывать при добавлении нового канала, а классы каналов не нужно изменять при добавлении нового типа уведомления.

## Зачем это нужно

Использование `Bridge` делает программу более гибкой и расширяемой.

Преимущества такого подхода:

- уменьшается количество классов для комбинаций
- код становится понятнее и логичнее по структуре
- можно независимо добавлять новые типы уведомлений
- можно независимо добавлять новые способы отправки
- снижается связность между частями программы

## Итог

В этой программе паттерн `Bridge` используется для разделения двух независимых измерений: видов уведомлений и способов их отправки. Благодаря этому любое уведомление можно отправить через любой канал, а программа остается удобной для расширения и сопровождения.
